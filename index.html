The Context Problem

Context windows are finite. Projects aren't.

Naive Approach

Dump everything in context.
Token waste. "Lost in the middle" effect.
Performance degrades at 50K+ tokens.

Smart Approach

Retrieve exactly what's relevant.
Module spec, domain policies, dependencies.
But how do you know what's relevant?

LLM output quality is bounded by context quality. Give an LLM the wrong context, and it produces confident, wrong code.

~60 seconds. "Here's the fundamental challenge: an agent needs to implement work item 42. What context does it need? The module spec? Domain policies? Prior review findings? Dependency interfaces? Context windows are finite, but projects keep growing. If you dump everything in, you waste tokens and the LLM gets confused. But if you're too selective, you miss critical information. You need a principled way to rank every artifact by relevance." BREADCRUMB 4: "This is exactly the problem that DDD bounded contexts solved partially for me — they reduced the scope. But I still needed to rank WITHIN the scope. Which policies matter most for this task? That's what drove me to PPR."

First Attempt: Breadth-First Search

Start at the work item, walk N hops, collect everything

▸  Simple — walk 2-3 hops, include every node you reach

▸  Hard cutoff — hop 3 is in, hop 4 is out. No ranking.

▸  Hub explosion — popular policies drag in every work item that references them

▸  No weighting — a critical dependency and a loose reference get equal treatment

On ideate's graph:
BFS depth 2 → ~40-60 nodes
BFS depth 3 → 100+ nodes
Most are irrelevant. No way to rank them.

~60 seconds. "BFS was the obvious first approach. Walk N hops from the work item and collect everything. Simple, and it works for small projects. But on ideate's own graph, depth 2 returns 40-60 nodes and depth 3 returns over a hundred. Most are irrelevant, and you have no way to rank them. You end up stuffing everything into context — exactly the problem we were trying to solve."

Random Walks

What if the graph could rank the nodes for us?

Instead of collecting everything within N hops, follow links at random from your starting node — thousands of times.

Nodes you visit more often are more relevant. Nodes reachable by many short paths score higher than nodes reachable by one long path.

This is PageRank (Brin & Page, 1998) — the algorithm that ranked the web. The same math works on any graph, including a graph of SDLC artifacts.

vs. BFS: no hard cutoff. Relevance decays smoothly with distance. A distant node reachable by many paths scores higher than a nearby node with one weak connection.

~60 seconds. "Here's the insight. Instead of grabbing everything within N hops, imagine following links at random from your starting node — thousands of times. Count how often you visit each node. That visit frequency IS the relevance score. This is PageRank — the same algorithm that ranked the web. And the critical difference from BFS: no hard cutoff. Relevance decays smoothly. A distant node that's reachable by many paths can outrank a nearby node with one weak connection."

Personalized PageRank

From global importance to local relevance

Standard PageRank: random walk jumps back to any node → global ranking

Personalized PageRank: random walk jumps back to a seed node with probability $\alpha$ → ranking relative to seed

For ideate: seed = the work item being executed → ranks every artifact by relevance to this task

The walk always returns to the starting point. Nearby nodes score higher.
But structurally important distant nodes still surface.

~75 seconds. "Here's the key modification. In standard PageRank, the random walk can jump back to any node in the graph — giving you global importance. In Personalized PageRank, it always jumps back to a specific seed node — in our case, the work item we're building. This biases the entire ranking toward the seed's neighborhood. Nodes close to the work item score higher. But structurally important nodes that are farther away — like a frequently-referenced domain policy — still surface. PPR finds relevance, not just proximity."

The Math

$\vec{\pi}$ gives us the relevance score for every node in the graph:

$$\vec{\pi} = \alpha \vec{s} + (1 - \alpha) M \vec{\pi}$$

$\vec{\pi}$ — relevance scores over all nodes (what we want)

$\vec{s}$ — seed vector (1.0 on the work item, 0 elsewhere)

$M$ — transition matrix (edge weights, column-normalized)

$\alpha$ — jump-back probability (typically 0.15 — controls exploration depth)

Iterate until convergence. ~20 iterations. One computation pass. No LLM calls for retrieval.

~90 seconds. Build the formula term by term. "This is the core equation. Pi is the vector of relevance scores — it tells us how relevant each artifact is to the task at hand. s is the seed vector — all the starting weight is on our work item. M is the transition matrix — it encodes the graph structure with edge weights. And alpha controls how often the walk jumps back to the seed instead of following an edge. Iterate this equation about 20 times and it converges to stable scores. Every node now has a relevance score. No LLM calls needed. Pure math."

Why Not BFS?

A breadth-first search walks N hops and treats every edge equally. It can't distinguish a critical dependency from a loose reference.

With weighted edges, PPR naturally emphasizes dependencies during execution, policies during review, and research during planning — same graph, different context for each phase.

~75 seconds. "The transition matrix isn't uniform. Different edge types carry different weights depending on what you're doing. BFS treats every edge the same and gives you a hard cutoff at N hops — everything inside is in, everything outside is out. PPR with weighted edges naturally emphasizes what matters for each phase. When executing, dependencies get full weight. When reviewing, findings and policies dominate. Same graph, different walks, different context."

What This Costs

PPR is cheap. The alternative isn't.

Iterative Retrieval

Ask an LLM to retrieve context, read it, decide if it needs more, retrieve again, repeat.

Each iteration costs tokens. Multi-hop questions require 3-5 iterations. At scale, this dominates your API bill.

PPR

One matrix computation. ~20 iterations of math, not LLM calls. Runs in milliseconds on SQLite.

HippoRAG (NeurIPS 2024) compared these approaches:
10-20x cheaper — one math pass vs. multiple LLM calls
6-13x faster — milliseconds vs. seconds per retrieval
Same quality on multi-hop retrieval benchmarks

~60 seconds. "Here's why PPR matters practically, not just theoretically. The alternative to PPR is iterative retrieval — you ask an LLM to find relevant context, it reads it, decides it needs more, retrieves again. Each iteration costs tokens. For multi-hop questions — where the answer requires connecting artifacts 2-3 hops apart — you need 3-5 iterations. That dominates your API bill. PPR replaces all of that with one matrix computation. About 20 iterations of math, not LLM calls. It runs in milliseconds on SQLite. HippoRAG at NeurIPS 2024 measured this directly: PPR-based retrieval is 10-20x cheaper and 6-13x faster, with comparable quality on multi-hop benchmarks." BREADCRUMB 5: "This is where the cost problem forced me to go. When your API bill makes you rethink everything, you either quit or you do the math."

Empirically calibrated — edge weights tunable from actual usage data, not intuition
